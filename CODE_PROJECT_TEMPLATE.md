# Code Project Template

**Purpose:** Every agent-generated code project follows this structure. Consistent layout means you build muscle memory вЂ” same folders, same conventions, same build commands across all topics. Interview-ready, runnable, dockerized.

**When to use:** Triggered by `deep dive`, `build`, `implement`, `scaffold`, `create project` in Slack (usually alongside the design doc generation).

**Output location:** `workspace-{agent}/code/{YYYY-MM-DD}-{topic-slug}-demo/`

### Naming conventions
- **Package**: `com.openclaw.{topic_slug_snake_case}` (e.g., `com.openclaw.saga_orders`)
- **Main class**: `{TopicSlugPascalCase}Application.java`
- **Config properties prefix**: `openclaw.{topic-slug}.*`
- **Docker image**: `openclaw/{topic-slug}:latest`

### pom.xml conventions
- Parent: `spring-boot-starter-parent` latest 3.3.x
- Java version: 21
- Required dependencies: `spring-boot-starter-actuator`, `spring-boot-starter-validation`
- Test dependencies: `spring-boot-starter-test`, `testcontainers-{mongodb|kafka|postgresql}`
- Build plugin: `jib-maven-plugin` for OCI-compliant images OR `spring-boot-maven-plugin` for fat JAR

## Required Files in EVERY Project

### 1. `README.md` вЂ” minimum required sections

```markdown
# {Project Title}

{One-paragraph description matching the design doc TL;DR.}

## Links
- Design doc: `../../docs/{YYYY-MM-DD}-{topic-slug}.md`
- Postman: `./postman/`
- PR: {PR URL}

## Quick Start

\```bash
docker-compose up
\```

App runs at http://localhost:8080
Actuator at http://localhost:8080/actuator/health

## Prerequisites
- Java 21 / Node 20 / Docker Desktop or OrbStack
- {Any topic-specific tools}

## What this project demonstrates
- {Pattern 1}
- {Pattern 2}
- {Pattern 3}

## How to test
- Unit tests: `mvn test` / `npm test`
- Integration tests: `mvn verify -P integration-test`
- E2E: `./postman/newman-run.sh` or `npx playwright test`

## Cleanup
\```bash
docker-compose down -v
\```
```

### 2. `docker-compose.yml` вЂ” must include
- All external dependencies (Kafka, Mongo, Postgres, Redis, etc.)
- The app itself (build from Dockerfile)
- Healthchecks for every service
- Named volumes for persistence
- Explicit versions (no `:latest`)

### 3. `.env.example` вЂ” all env vars referenced in the code
- Never commit a real `.env` вЂ” add to `.gitignore`
- All secrets use `${VAR_NAME}` syntax in `application.yml` / config files

### 4. `Dockerfile` вЂ” multi-stage build
- Stage 1: Build (maven/gradle/npm with full JDK/node)
- Stage 2: Runtime (distroless for Java, nginx for React)
- Non-root user
- Healthcheck instruction

## Standards (All Projects)

### Security
- вќЊ Never hardcode secrets вЂ” always env vars
- вќЊ Never commit `.env`, `.key`, `.pem`, `credentials.json`
- вњ… Use `.env.example` to document required vars
- вњ… Validate all inputs (Bean Validation for Java, Zod for React)
- вњ… Sanitize logs (no PII, no tokens)

### Observability
- вњ… Spring Boot Actuator endpoints exposed (`/actuator/health`, `/actuator/info`, `/actuator/metrics`)
- вњ… Structured logging (JSON format for production)
- вњ… Correlation IDs propagated across services

### Testability
- вњ… Testcontainers for integration tests вЂ” no H2 / in-memory databases
- вњ… At least one happy path + one failure path integration test
- вњ… No `Thread.sleep()` вЂ” use `Awaitility` for async assertions

### Code quality
- вњ… No `// TODO` without a GitHub issue link
- вњ… Public methods have Javadoc / TSDoc
- вњ… Feature flags for Phase 2 features

## Notes for Agents

1. **Projects must run out of the box.** If `docker-compose up` doesn't start the full stack successfully, the project is incomplete.
2. **Keep it minimal.** Phase 1 of the design doc = what this project implements. Phase 2 goes to issues/TODOs.
3. **One topic = one project.** Don't bundle unrelated patterns.
4. **Reference the design doc.** Add `see docs/{topic}.md` comments in tricky code sections.
5. **Delete generated README boilerplate.** The README must be specific to the topic, not a generic scaffold.
